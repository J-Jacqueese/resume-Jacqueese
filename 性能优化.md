# 性能优化 

## 当在浏览器中输入Google.com 并且按下回车之后发生了什么？

1. **URL解析**：输入的是合法的URL还是关键词搜索（检查用户输入、beforeunload 事件、浏览器进入加载状态）

2. **缓存判断**：浏览器会判断所请求的资源是否在缓存里，如果在缓存里并且没有失效，那么就直接使用，否则向服务器发起新的请求。

3. DNS解析

    ：获取到域名对应的目标服务器IP地址(域名映射IP)

    - 浏览器缓存的DNS
    - 操作系统的DNS
    - 本地hosts域名服务器的DNS `递归请求(用户只需要发一次请求)`
    - 其他域名服务器的DNS：`迭代请求(用户根据各级域名服务器结果发多次请求)`
    - 根域名服务器
    - 顶级域名服务器
    - 权威域名服务器

4. **TCP连接**：三次握手、建立传输通信，浏览器会向服务器发送一个 SYN 请求，服务器收到请求后返回一个 SYN-ACK 响应，最后浏览器再发送一个 ACK 响应，完成三次握手，建立 TCP 连接。

5. **发起http请求**：浏览器向服务器发起HTTP请求,*特殊请求报文 keep-alive https http2 先进先出 队头阻塞*

6. **响应请求**：服务器响应HTTP请求，并返回响应消息HTML

7. **页面渲染**：对资源进行解析，再进行页面的渲染 。

    HTML-DOM CSS-CSSOM 渲染树 减少重溯和回流-重绘和重排 js解析 事件循环（准备渲染进程、提交文档、更新浏览器界面状态）

    - 准备渲染进程：浏览器进程检查当前 URL 是否和之前打开的渲染进程根域名是否相同，如果相同则复用原来的进程，如果不同则开启新的渲染进程。
    - 提交文档：浏览器会发出`提交文档`的消息给渲染进程，渲染进程接收到消息后，会和网络进程建立传输数据的`管道`，文档数据传输完成之后，渲染进程会返回`确认提交`的消息给浏览器进程；（文档指的是 URL 请求的响应体）
    - 注意：网络进程向渲染进程提交文档，是边下载边传递给渲染进程解析的，接收到第一批数据，便开始做 DOM 解析了！即提交文档后便开始进行解析 DOM，解析 CSS，生成布局树，绘制等操作。
    - 更新浏览器界面状态：浏览器进程在收到`确认提交`的消息后，会更新浏览器界面状态，包括了安全状态、地址栏的 URL、前进后退的历史状态，并更新 Web 页面。此时的 web 页面是空白页。
    - 渲染文档：渲染进程对文档进行页面解析和子资源加载，HTML 通过 HTML 解析器转成 DOM Tree，CSS 按照 CSS 规则和 CSS解释器转成 CSSOM TREE，两个 tree 结合，形成 render tree（不包含HTML的具体元素和元素要画的具体位置），通过 Layout 可以计算出每个元素具体的宽高颜色位置，结合起来，开始绘制，最后显示在屏幕中新页面显示出来。这期间牵扯到浏览器渲染流程，具体翻阅 [浏览器渲染](https://jonny-wei.github.io/blog/base/browser/browser-render)

8. **关闭TCP连接**：四次挥手关闭TCP连接

## 浏览器中的页面如何渲染的？（HTML文档的渲染过程）

1. HTML解析生成DOM树
2. CSS解析生成CSSOM树
3. DOM树和CSSOM树合成 构建渲染树
4. 根据渲染树树和视口进行Layout布局，计算每个节点的位置和大小(布局render树)
5. 绘制：将每个节点绘制到页面上(绘制render树，像素px信息)
6. 转交给合成进程，光栅化线程池，GPU进程合成，显示在屏幕上

### 渲染（FP、FCP）

- **FP** **(First Paint)(首次绘制)**：**首次渲染**的时间点。用于记录页面第一次绘制像素的时间。 FP 时间点之前，用户看到的都是**没有任何内容的白色屏幕**，可以视为**白屏时间**。
- **FCP (First Content Paint)(首次内容绘制)**：**首次有内容渲染**的时间点。用于记录页面首次绘制文本、图片、非空白 Canvas 或 SVG 的时间。在用户访问 Web 网页的过程中，FCP 时间点之前，用户看到的都是**没有任何实际内容的屏幕**。FCP 反映当前 **Web 页面的网络加载性能情况、页面 DOM 结构复杂度情况**、`inline script` 的执行效率的情况。
- 加载时间：页面运行时各个阶段的加载时间；
- TTFB (Time To First Byte)(首字节时间)：浏览器发起第一个请求到数据返回第一个字节所消耗的时间，这个时间包含了网络请求时间、后端处理时间；

### 渲染主要内容（FMP&LCP&SI）

- **LCP (Largest Contentful Paint)**：**最大的内容在可视区域内变得可见**的时间点。用于记录视窗内最大的元素绘制的时间，该时间会随着页面渲染变化而变化，因为页面中的最大元素在渲染过程中可能会发生改变，另外该指标会在用户第一次交互后停止记录。页面的最大内容（通常是比较核心的内容）加载完成的时间，这个最大内容可以是图片/文本块。它是一个 **SEO 相关的指标**。**官方推荐，在 2.5 秒内表示体验优秀。**
- FMP (First Meaningful paint)(首次有意义绘制)：首次有意义绘制的时间点是页面可用性的量度标准；（计算复杂）
- SI（Speed Index）: 衡量**页面可视区域加载速度**，帮助检测页面的加载体验差异。（SI 因为计算复杂，指标难以解释）

### 交互（TTI & TBT）

- TTI（Time to Interactive）: 测量页面**从开始加载到主要子资源完成渲染**，并能够快速、可靠地响应用户输入所需的时间。**TTI 值越小，代表用户可以更早地操作页面**，用户体验就更好。
- **TBT（Total Blocking Time）**: 阻塞总时间。**量化主线程在空闲之前的繁忙程度，有助于理解在加载期间，页面无法响应用户输入的时间有多久。**一个页面的 TBT，是**从 FCP 到 TTI 之间所有长任务的阻塞时间的总和。**

#### 交互时是否有延迟 - FID & MPFID

- **FID (First Input Delay)(首次输入延迟)**：用户首次和页面交互到页面响应交互的时间；实际能够**开始处理事件处理程序所经过的时间**，即**交互延迟时间**。
- MPFID（Max Potential First Input Delay）：记录在页面加载过程中用户和页面进行首次交互操作可能花费的最长时间。
- 分割长任务、最小化主线程工作
- 优化页面，尽快做好交互准备
    - 移除关键路径上非必要组件加载的脚本
    - 尽最大限度地减少需要在客户端进行后处理的数据量；
    - 按需加载第三方代码
    - 优先加载可以为用户提供最大价值的内容
- 使用 Web Worker
- 减少 JS 执行时间
- 减小请求数量和请求文件大小

### 页面是否稳定 - CLS

- **CLS（Cumulative Layout Shift）**: **累积布局偏移**。量化了**在页面加载期间，视口中元素的移动程度**。在一个页面的生命周期中，会不断的发生布局变化（layout shift），对每一次布局变化做一个累积的记分，其中得分最大布局变化即为 CLS，是**衡量页面稳定性**的重要指标。

## 打包方面优化

**减少产物体积**：`代码压缩、代码分割、摇树优化、优化 SourceMap、按需引入、路由懒加载、动态垫片、依赖外置、作用提升等`

**减少打包时间**：缩减范围、定向搜索、持久化缓存、提前构建、按需构建、并行构建、可视结构等

### 减小产物体积，提高首屏性能

- **`压缩代码`**。利用 AST 压缩混淆代码，压缩的过程中还伴随着 tree-shaking。目的是减小产物体积，减小页面加载和渲染时间，防止反向编译工程的可能性，进而提高首屏性能。

    - **压缩 HTML**：**HtmlWebpackPlugin** 插入引用，根据模版生成 HTML，通过 `HtmlMinifierTerser` 压缩 HTML。
    - **压缩 CSS**：**CssMinimizerWebpackPlugin**
        - **分离 CSS 样式，成单独 bundle**。避免大量 CSS 内联 JS，减小 JS Bundle 体积与样式闪动，提高资源下载速度；**利用浏览器强缓存，减少静态资源的网络请求**，进而提高首屏性能。
    - **压缩 Javascript**：[Terser](https://link.juejin.cn/?target=https%3A%2F%2Fgithub.com%2Fterser%2Fterser)：terser-webpack-plugin（ES6 压缩）、Webpack5.0 默认开启简单用法只需通过 optimization.minimize 配置项开启压缩功能即可。
    - **压缩目的：**
        - 压缩混淆代码
        - 降低浏览加载资源体积
        - 降低页面渲染时间
        - 防止反向编译工程的可能性

- **按需引入**。组件库、工具库按需引入，减小产物体积，进而提高首屏性能。

    - babel-plugin-import
    - babel-plugin-component

- **`利用 Tree-Shaking`**。始终使用 ESM（模块依赖是高度确定的 `基于 ES Module 进行静态分析，通过 AST 将用不到的函数进行移除，从而减小打包体积。`），移除 JS 中 未使用的文件（只用到了叶子的某些 js 文件）、删除无用的代码。利用 Tree-Shaking，减小产物体积，进而提高首屏性能。（loadash 只用用到的）

    - Tree-Shaking 的实现大致上可以分为三个步骤：
        - 「构建」阶段，「收集」 模块导出变量并记录到模块依赖关系图 ModuleGraph 对象中；
        - 「封装」阶段，遍历所有模块，「标记」 模块导出变量有没有被使用；
        - 使用代码优化插件 —— 如 Terser，「删除」无效导出代码。

- **`使用 CDN 加速`**。 在构建过程中，**将引⽤的静态资源路径修改为CDN上对应的路径**。可以利⽤webpack对于 output 参数和各loader的 publicPath 参数来修改资源路径

    - webpack 的图片优化策略

        - 使用 `url-loader` 内联资源，将小图像转为 base64 形式的字符串，从而减少 HTTP 请求，但是它会增加打包结果的尺寸，生产环境可能不太适合。[url-loader](https://link.juejin.cn/?target=https%3A%2F%2Fv4.webpack.js.org%2Floaders%2Furl-loader%2F)：有两种表现，对于小于阈值 `limit` 的图像直接转化为 base64 编码；大于阈值的图像则调用 `file-loader` 进行加载。`url-loader` 同样适用于大多数图片格式，且能将许多细小的图片直接内嵌进产物中，减少页面运行时需要发出的网络请求数，**在 HTTP 1.1** 及之前版本中能带来正向的性能收益。

        - 使用 `file-loader` 对其延迟加载同时为较大的文件生成单独的请求地址。file-loader 能够将导入的图片存放到特定的目录，并返回图片的路径地址。此技术适用于其他类型的资源，例如字体。[file-loader (opens new window)](https://link.juejin.cn/?target=https%3A%2F%2Fv4.webpack.js.org%2Floaders%2Ffile-loader%2F)：将图像引用转换为 url 语句并生成相应图片文件，经过 `file-loader` 处理后，原始图片会被重命名并复制到产物文件夹，同时在代码中插入图片 URL 地址。

        - raw-loader：不做任何转译，只是简单将文件内容复制到产物中，适用于 **SVG 场景**。经过 raw-loader 处理后，SVG 资源会被直接复制成字符串形式。

            - [svg-inline-loader](https://link.juejin.cn/?target=https%3A%2F%2Fwww.npmjs.com%2Fpackage%2Fsvg-inline-loader)：能够自动删除 SVG 图片中与显式无关的各种原信息，达到压缩效果；
            - [svg-url-loader](https://link.juejin.cn/?target=https%3A%2F%2Fwww.npmjs.com%2Fpackage%2Fsvg-url-loader)：以 [DataURL (opens new window)](https://link.juejin.cn/?target=https%3A%2F%2Fdeveloper.mozilla.org%2Fzh-CN%2Fdocs%2FWeb%2FHTTP%2FBasics_of_HTTP%2FData_URIs)方式导入 SVG 图片，相比于 Base64 更节省空间；
            - [react-svg-loader](https://link.juejin.cn/?target=https%3A%2F%2Fwww.npmjs.com%2Fpackage%2Freact-svg-loader)：导入 SVG 图片并自动转化为 React 组件形态，效果类似 [@svgr/webpack](https://link.juejin.cn/?target=https%3A%2F%2Fwww.npmjs.com%2Fpackage%2F%40svgr%2Fwebpack)；

        - svg 图片使用 `svg-sprite-loader`、`svg-url-loader` 等处理。前者将小的 SVG 文件合并为一个 sprite 文件，从而可以更有效地加载，因为您可以避免请求开销。后者将 SVG 加载为 UTF-8 编码的数据 URL。结果比 Base64 更小，解析更快。

        - 使用 `image-webpack-loader`、`svgo-loader`（专用于 SVG）或 `imagemin-webpack-plugin` 压缩图片，减少图片下载所需带宽量。

        - 使用图片占位符。`image-trace-loader` 加载图像并将结果编码为 image/svg+xml 格式的 URL 数据。它可以于 file-loader 和 url-loader 一起使用，以便在加载实际图像时显示占位符。

        - 利用 srcset。`resize-image-loader` 和 `responsevie-loader` 允许您为现代浏览器生成 srcset 属性所需要的图像集合。srcset 属性可以让浏览器决定加载哪些图像以及何时提高性能。

        - 加载 Sprite(雪碧图)。Spriting 技术允许您将多个较小的图像组合成单个图像。它已经被用于游戏中的动画，但对于 Web 开发也很有价值，因为它节省了很多请求开销。`webpack-spritesmith` 生成雪碧图。

        - **使用 WebP 格式的图片。**谷歌公司开发的一种旨在加快图片加载速度的图片格式。图片压缩体积大约只有JPEG的2/3，并能节省大量的服务器带宽资源和数据空间。Facebook、Ebay等知名网站已经开始测试并使用WebP格式。

            - 一、核心思路（一句话总括）

                - **优先提供 WebP，浏览器支持则使用；不支持自动回退到 JPEG/PNG。**通过 `HTML 的 <picture>、CSS 回退、以及服务端/CDN 的内容协商来实现，并用 Vary 头避免缓存串扰。`

                二、前端实现

                1. HTML（首选 <picture>）

                - 原理：按类型匹配选择最优资源，不支持则回退到 <img>。

                1. CSS 背景（提供回退）@supports/ image-set

                1. JS 检测（备选，不建议首选）

                - 缺点：首屏切换会闪烁；仅在无法改模板时使用。
                    const canWebp = document.createElement('canvas').toDataURL('image/webp').startsWith('data:image/webp');

                三、服务端/CDN 自适应（免改模版方案）

                - 原理：基于请求头 Accept 中的 image/webp 来选择返回 WebP 或回退格式。
                - 关键：返回时设置 Vary: Accept，避免 CDN/代理把 WebP 缓存给不支持的客户端。
                - CDN：开启“图片格式自适应/图片优化”，同时输出正确 Content-Type。

                四、**构建与产出策略**

                - **为每张源图产出多格式**：.avif（可选）/.webp + .jpg/.png；并生成多尺寸供响应式加载。
                - 常用工具：sharp、imagemin、Squoosh CLI；集成到 webpack/vite 构建流程。

    在 Webpack 生态中有不少优秀的图像压缩组件[image-webpack-loader](https://link.juejin.cn/?target=https%3A%2F%2Fgithub.com%2Ftcoopman%2Fimage-webpack-loader) 组件功能齐全且用法简单，更推荐使用。

- **`优化 SourceMap`**。生产环境用 `cheap-module-source-map`配置项，减少 SourceMap 体积，进而减小产物体积，进而提高首屏性能

    开发环境推荐： cheap-module-eval-source-map

    **生产环境推荐： cheap-module-source-map**

    一般开发环境开启，线上环境关闭，线上排查问题时可将 sourcemap 上传错误监控系统。

    - eval：使用 eval 包裹模块代码，包裹最后有 sourcemap 信息
    - source map：产生 `.map` 文件，包括行列信息，loader 也有对应的 sourcemap
    - cheap：不包含列信息，只有行信息，没有列信息，loader 也没有对应的 sourcemap，对应的都是 loader 转换后的代码，不是纯正的源代码
    - inline：将 `.map` 作为 DataURL 嵌入，不单独生成 `.map` 文件，不推荐使用，因为这样会造成源代码体积巨大
    - module：包含 loader 的 sourcemap

    (inline-)(cheap-)(module-)(eval-)source-map 按序组合

- **路由懒加载。懒加载的本质实际上就是代码分离。把代码分离到不同的 bundle 中**，然后按需加载或并行加载这些文件。单页面应用，可能会有很多的路由引入，打包后产物很大。**Webpack 异步加载，将代码分割成单独 Chunks，当代码运行到需要它们的时候再进行加载。**

    - import() 语法实现动态导入。
    - Babel 本身不支持动态 import 语法，它需要 `@babel/plugin-syntax-dynamic-import` 配合才能工作。

- **Code Split 代码分割**。根据产物包的体积、引用次数等做分包优化，减少产物体积，有效利用浏览器缓存，提高首屏性能。

    **最佳分包策略**，包括：

    - 针对 `node_modules` 资源：
        - 可以将 `node_modules` 模块打包成单独文件(通过 `cacheGroups` 实现)，防止业务代码的变更影响 NPM 包缓存，同时建议通过 `maxSize` 设定阈值，防止 vendor 包体过大；
        - 更激进的，如果生产环境已经部署 HTTP2/3 一类高性能网络协议，甚至可以考虑将每一个 NPM 包都打包成单独文件。
    - 针对业务代码：
        - 设置 `common` 分组，通过 `minChunks` 配置项将使用率较高的资源合并为 Common 资源；
        - 首屏用不上的代码，尽量以异步方式引入；
        - 设置 `optimization.runtimeChunk` 为 `true`，将运行时代码拆分为独立资源。

    实践中，**分包策略的好坏直接影响应用的运行性能，常用策略一是单独打包 `node_modules` 代码(习惯称为 `vendor`)，二是单独打包被频繁使用的模块(习惯称为 `common`)。**

    - 使用 html-webpack-externals-plugin 基础库分离，通过 CDN 引入，不打入 bundle 中。
    - 使用 SplitChunksPlugin 不仅可以分离基础包，还可以提取公共带代码(推荐)。
    - 使用 DLLPlugin 预编译资源模块分离基础包(内置插件，更好的分包)
    - 使用 hard-source-webpack-plugin 是 DLL 的更好替代者

## 渲染优化

### 避免重排与重绘

避免 DOM 的回流。也就是尽量避免重排和重绘操作。

#### 如何避免重绘重排重绘

- 操作DOM时，尽量在低层级的DOM节点进行操作
- 不要使用`table`布局， 一个小的改动可能会使整个`table`进行重新布局
- `避免使用CSS的表达式`
- 不要频繁操作元素的样式，对于静态页面，可以修改类名，而不是样式。
- （动画优化）使用absolute或者fixed，使元素脱离文档流，这样他们发生变化就不会影响其他元素
- `避免频繁操作DOM`，可以创建一个文档片段`documentFragment`，在它上面应用所有DOM操作，最后再把它添加到文档中
    - 插入的不是 DocumentFragment 自身，而是它的所有子孙节点。将DocumentFragment 节点插入DOM树时，不会触发页面的重绘，这样就大大提高了页面的性能。
- DOM 离线化：将元素先设置`display: none`，操作结束后再把它显示出来。因为在display属性为none的元素上进行的DOM操作不会引发回流和重绘。
    -  (**`display: none`**) 将元素从渲染树中完全移除，元素既不可见，也不是布局的组成部分，之后在该 DOM 上的操作不会触发回流与重绘，操作完之后再将`display`属性改为显示，只会触发这一次回流与重绘。
- 将DOM的多个读操作（或者写操作）放在一起，而不是读写操作穿插着写。这得益于**浏览器的渲染队列机制**。
    - **浏览器会将所有的回流、重绘的操作放在一个队列中，当队列中的操作到了一定的数量或者到了一定的时间间隔，浏览器就会对队列进行批处理。这样就会让多次的回流、重绘变成一次回流重绘。**

#### 重排

- 定义

    当**通过 `JS 或者 CSS 修改元素的几何属性`**，例如改变元素的宽度、高度等，那么浏览器会**触发重新布局**，解析之后的一系列子阶段，这个过程就叫重排。

- `触发条件`：页面的首次渲染、浏览器的窗口大小发生变化、元素的尺寸或者位置发生变化、查询某些属性或者调用某些方法、添加或者删除可见的DOM元素

- `检测方式`

    - Performance 面板（最全面）
        - 录制性能：查看 Main 线程的事件火焰图，关注以下阶段与事件：
            - Layout（布局）：显示布局计算耗时、影响的节点数量
            - Style Recalculation：样式重计算耗时
            - Paint/Rasterize：绘制与栅格化耗时
            - Composite Layers：合成耗时
        - Layout Shift（CLS 相关）：可视化布局偏移，识别加载过程中的位置跳动
        - FPS/CPU/堆栈：找到触发的具体调用栈（谁在读写样式）
    - Rendering 辅助叠层
        - Paint flashing（勾选 Paint flashing）：页面发生重绘时会闪绿色高亮，便于直观看到重绘范围与频率
        - Layout Shift Regions：高亮布局偏移区域
        - FPS meter：帧率监控，识别卡顿
    - Layers/TODO: Elements 面板
        - Layers 视图查看层树，识别哪些元素被提升为合成层
        - 在 Elements 面板查看 computed 样式变化、检查 will-change 是否生效

#### 重绘

- 定义

    当通过`JS或者 CSS 修改元素的绘制属性`，例如改变元素的背景颜色，那么布局阶段将不会被执行，因为并没有引起几何位置的变换，所以就直接进入了**绘制阶段（即生成待绘制列表）**，然后执行之后的一系列子阶段，这个过程就叫重绘。

- 触发条件：color、 background 相关属性(background, background-color, background-image, background-position,background-repeat, background-size)、outline 相关属性(outline, outline-color, outline-style)： text-decoration、border-radius、 visibility、 box-shadow、`transform`

- 优化方案

    - 合并多次操作

### defer 与 async 的区别

没有 defer（**延迟执行引入的 js**） 或 async（**异步执行引入的 js**）,浏览器会立即加载并执行指定的脚本，“立即” 指的是在渲染该 script 标签之下的文档元素之前，也就是说不等待后续载入的文档元素，读到就加载并执行。

- 脚本与DOM/其它脚本的依赖关系很强：对 `<script>` 设置defer
- 脚本与DOM/其它脚本的依赖关系不强：对 `<script>` 设置async

- **执行时机**：
    - `async`脚本在下载完成后会尽快执行。
    - `defer`脚本则会等到文档解析完成后按顺序执行。

- **执行顺序**：
    -  `async`不保证脚本的执行顺序。
    -  `defer`保证了脚本的执行顺序与它们在文档中出现的顺序相同。

- **阻塞渲染**：
    - `async`和`defer`脚本都不会阻塞页面渲染，它们都是异步加载的。

- **兼容性**：
    - `defer`属性在现代浏览器中得到了很好的支持，但`async`属性更古老，支持更广泛。

### 白屏渲染优化/CSR和SSR的区别

- Web 前端诞生了 SSR 、CSR、预渲染等技术。白屏出现的原因是因为要等待文件加载、CSSOM 构建、JS 解析等过程，而这些过程比较耗时，导致用户会长时间出于不可交互的首屏灰白屏状态，从而给用户一种网页很“慢”的感觉。
- **CSR（客户端渲染）**：前端自己下载 JS，自己渲染页面 → **先白屏，后出内容**
- **SSR（服务端渲染）**：服务器直接把**现成的 HTML 页面**发给你 → **打开就有内容**

### 服务端渲染 SSR/注水

服务端渲染（SSR），服务端直出页面（SPA->完整的 HTMl 字符串）返回给客户端这个过程就叫做服务端渲染。

- 基本工作流程

    - **用户请求**：浏览器向服务器发起页面请求。
    - **服务器处理**：
        - 数据获取：服务器（可能）需要先异步获取该页面所需的数据。
        - **渲染HTML**：在服务器端执行JavaScript代码，将组件渲染为HTML字符串。

    - **响应HTML**：服务器将生成的完整HTML以及可能需要的初始数据（脱水数据）一并发送给浏览器。

    - **浏览器渲染**：浏览器接收到HTML后，立即解析并显示页面内容

    - `注水 （Hydration）`：JS代码接管静态HTML，**添加事件监听器**、恢复应用状态等 使页面变得可交互 。

        - **核心步骤**
            - `获取HTML 骨架`：浏览器收到 SSR返回的HTML。
            - `加载 JavaScript`：浏览器下载React 和应用的JS包。
            - `“注水”过程`：
                -  React 在内存中重新构建虚拟 DOM （Virtual DOM）。
                -  遍历DOM树，将VDOM 节点与真实 DOM 节点一一对应。
                -  附加事件监听器（如 onClick）。

        - **痛点一**：`全有/全无`
            - 问题：必须等待所有组件的 JavaScript代码都下载完毕，才能对整个应用进行 Hydration。
            - 后果：**只要有一个组件加载缓慢**，就会阻塞整个页面的交互。
        - **痛点二**：`同步阻塞`
            - 问题：Hydration 是一个`同步的、CPU 密集型`的过程。
            - 后果：
                - 在 Hydration 期间，浏览器主线程被占用。
                - 用户无法进行任何操作（点击、输入等），页面假死。
                - 糟糕的 TTI（Time to Interactive）体验。
        - React18 优化（打破全有/全无）
            - 流式服务端渲染（Streaming SSR）
                - 原理：
                    • 利用<Suspense> 将应用拆分为独立的块。
                    • 服务器不再需要等待所有数据准备好，而是先发送HTML“外壳”。
                    • 当某个 <Suspense>包裹的组件在服务器端准备好后，就将其 HTML“流式”传输到客户端。
            - 选择性注水（Selective Hydration）
                - 定义：即使JS代码还未完全下载，React 也可以对已经可见的部分进行Hydration。
                - 关键：Hydration 不再是一个整体的、同步的任务。
                - 智能事件记录：
                    - 问题：如果用户在组件 Hydration 之前就点击了它，怎么办？
                        - React 18 的方案：React会在根节点记录这个点击事件。当对应的组件完成 Hydration 后，React 会“重放”（replay）这个事件。确保用户交互不会丢失。
                - 场景：Comments 组件的JS 还在加载，但用户点击了 Header。
                    - React 17及之前：无响应。
                    - React 18：
                        - React 会优先对用户正在交互的 Header 组件进行 Hydration。
                        - Comments 的Hydration 会在后台继续，不阻塞用户操作。

- 优点

    - 更好的 SEO： SPA 需要发请求浏览器不会等，SSR 是直接由服务端返回已经渲染好的页面（数据已经包含在页面中），所以`搜索引擎爬取工具可以抓取渲染好的页面；`

    - 首屏加载更快（FCP/LCP）： SPA 会等待编译后的 js 文件都下载完成后，才开始进行页面的渲染，文件下载等需要一定的时间等，所以首屏渲染需要一定的时间；`SSR 直接由服务端渲染好页面直接返回显示，无需等待下载 js 文件及再去渲染等`，所以 SSR 有更快的内容到达时间；
        - **CSR(客户端渲染)**：浏览器下载HTML骨架->下载JS->JS执行-> 请求数据->渲染页面。
        - SSR：浏览器请求->服务器处理（数据＋渲染）-> 返回完整HTML->浏览器显示。


缺点

- 更高的服务器负载：
    - 页面渲染从客户端转移到服务器，增加了服务器的计算压力和响应时间。

- 更复杂的构建和部署：
    - 需要同时管理和维护服务端和客户端的代码。
    - Node.js环境通常是必需的。
- 开发复杂度提升：
    - 需要考虑组件在服务端和客户端的生命周期、数据获取方式等。
    - 某些浏览器特有的API（如 window）不能在多端直接使用。

- Time to Interactive 变长。

使用场景

- **内容型网站**：如博客、新闻门户、电商商品详情页，SEO至关重要。
- **对首屏加载速度有极致要求的应用**：提升用户留存和转化。
- **目标用户网络环境较差或设备性能不一**：确保基础体验。
- 可以不使用SSR的场景：
    - 管理后台、内部系统等对SEO无要求，且用户网络环境通常较好的应用。
    - 纯静态站点（可考虑SSG-静态站点生成）。

    - 定义：将代码（HTML, CSS, JavaScript）转换为用户在浏览器中看到的交互式网页的过程。

        - 性能：直接关系到网站的整体表现。

**SSR**（Next->getServerSideProps）

- 是什么：在每次请求时，服务器动态生成页面的完整 HTML，然后发送给客户端。
- 核心逻辑：请求->服务器处理->生成HTML->发送给客户端->客户端激活 （Hydration）。
- 优点：
    - SEO友好：搜索引擎可以直接抓取完整HTML内容。
    - 数据最新：每次请求都获取最新数据。
- 缺点：
    - 服务器压力：每次请求都需要服务器计算。
    - TTFB （Time to First Byte） 可能较慢：相比静态站点。

**SSG(静态站点生成)**

- 是什么：在构建时 （build time）就为每个页面生成静态HTML文件。
- 核心逻辑：构建->生成所有HTML页面->部署到CDN-> 用户请求->直接从CDN获取HTML。
- 优点：
    - 极速加载：HTML已生成，可被CDN缓存，TTFB极低。
    - 服务器压力小：只需提供静态文件。
    - SEO友好：搜索引擎可直接抓取。
- 缺点：
    - 数据非实时：内容更新需要重新构建整个站点（或部分）。
    - 构建时间：站点页面很多时，构建时间可能较长。

**ISR**：增量静态再生（getStaticProps 增加热 validate函数）

- 是什么：SSG的增强版。在构建时生成静态页面，但允许在用户请求后，按需、增量地重新生成特定页面，而无需完全重新构建。
- 核心逻辑：
    - 初始构建时生成静态页面。
    - 用户请求-> 提供缓存的静态页面。
    - 同时（若配置了 revalidate 且时间已到），在后台重新生成页面。下次请求该页面时，提供更新后的版本。
- 优点：
    - 兼具SSG的速度和部分SSR的数据新鲜度。
    - 减少构建时间：无需每次都完整构建

### 预渲染

所谓预渲染，就是在项目的构建过程中，通过一些渲染机制，比如 puppeteer或则 jsdom 将页面在构建的过程中就渲染好，然后插入到 html 中，这样在页面启动之前首先看到的就是预渲染的页面了。

可以用 `prerender-spa-plugin` 做预渲染，这样就可以在浏览器进行渲染，而不需要将 Vue 或者 React 代码部署到服务器上。

预渲染的作用:

- 大型单页应用的性能瓶颈：JS下载+解析+执行
- SSR的主要问题：牺牲TTFB 请求过程，来补救首次加载 First Paint 实现起来，也很复杂
- Pre-rendering 打包时提前渲染页面，没有服务端参与

预渲染有什么缺点呢？

- 动态数据无法展示，不同的用户看到的都是同样的页面
- 路由很多时，代码构建时间太长
- 用户容易误操作，由于预渲染时 js 还没有加载，因此展示出来的内容没有 js 的交互逻辑，比如按钮点击，在 js 加载完成之前用户点击是没有反应的
- 预加载内容很少，当页面有内容是依赖动态数据加载时，在编译时是无法加载出动态数据的，因此会导致这部分内容编译不出来

### 骨架屏

骨架屏的实现原理和预加载类似，都是利用了 Puppeteer 爬取页面的功能，**Puppeteer 是 chrome 出的一个 headlessChromenode 库，提供了 API 可以抓取 SPA 并生成预渲染内容**，和预加载不太一样的是骨架屏技术会在 Puppeteer 生成内容后，利用算法将生成的内容进行替换，生成骨架页面，`page-skeleton-webpack-plugin` 是一个用来生成骨架屏的 webpack 插件。

### 代码分割：使用 React.lazy + Suspense 实现组件懒加载 

- 核心思想：将你的代码从单个巨大的包（bundle）拆分成多个小块（chunks）。
- 加载方式：这些小块可以按需加载或并行加载。
- 目的：只在用户需要时加载相应的代码。

- 优点：
    - 提升初始加载速度：用户首次访问时，只需下载运行首页所需的最少代码。
    - 改善应用性能：更小的代码包意味着更快的解析和执行时间。
    - 优化用户体验（UX）：减少等待时间，应用感觉更流畅、响应更 快
    - 节省带宽：尤其对移动端用户和网络环境较差的地区友好。

- 实现：
    - React.lazy()：动态导入组件。
    - <suspense>：处理懒加载组件的加载状态。
    - 优化初始包体积（Bundle Size），加快首屏速度。

- 最佳实践
    - 选择分割点：
        - 基于路由是常见且有效的起点。
        - 对于用户交互后才显示的UI（如模态框、复杂的非首屏组件）。
    - 加载状态 （Fallback UI）：
        - Suspense 提供了优雅的方案。设计好的加载指示器能提升体验。
    - 错误处理：
    - 动态导入可能会失败（网络问题等）。
    - 使用 Error Boundaries 包裹懒加载组件及其 Suspense，
    - 处理这些错误。
    - 避免过度分割：过多的分割可导致HTTP 请求开销增加。

### 虚拟列表化

点出问题（性能瓶颈）-> 提出方案（虚拟列表） -> 核心原理 -> 实现方法

- 性能瓶颈：
    - DOM节点过多：
        - 浏览器需要创建和维护大量DOM元素。
        - 内存占用显著增加。
        - 布局计算（Layout）、绘制（Paint）负担加重。 

    - React协调成本：
    - render 函数执行时间变长。
    - Diff算法在大量节点间比较，效率降低。
    - 频繁的更新可能导致掉帧，用户体验差。

- 只渲染视口内（Viewport）可见的列表项。（按需渲染、视觉欺骗）
- 为什么高效？
    - 大幅提升渲染性能：DOM节点数量维持在一个很小的量级，极大减轻浏览器负担。
    - 降低内存消耗：仅为可见项创建DOM和组件实例。
    - 更快的首次加载速度：应用启动时无需渲染全部数据。
    - 流畅的滚动体验：即使数据量巨大，滚动依然平滑。
- 如何实现？
    -  常用库：`react-window （推荐）,react-virtualized。`
- **核心原理**：`计算并仅渲染可见区域的Item。`
    - **容器** （Container）：具有固定高度和overflow:auto/scroll
    - **滚动内容区**（Scrollable Content）：一个高度等于所有列表项总高度的内部div，用于撑开滚动条。
    - **可视区域计算**：根据容器的 scro11Top、容器高度、以及每项的预估/固定高度，计算出当前可见的列表项的startIndex 和 endIndex。
    - **列表项渲染**：仅渲染 items.slice（startIndex, endIndex + 1）。
    - **绝对定位**：使用 position: absolute 和 top 样式将渲染的列表项精确放置到滚动内容区内的正确位置。
    - **滚动监听**：监听容器 scroll事件，重新计算可视区域并更新渲染。

如果列表项的高度是动态不固定的，虚拟列表如何处理？

-  react-virtualized 的 CelIMeasurer 可以用来动态测量和缓存行高。
-  react-window 提供了 VariablesizeList。（提供动态获取每一项函数）
-  手写：维护高度缓存数组，未渲染用预估高度，渲染后计算实际高度和后续项的位置 
-  可能需要预估高度，**渲染后获取真实高度再调整。**

虚拟列表有什么潜在的缺点或使用场景限制？

- 实现相对复杂（如果手写、键盘高度、计算差异）。
- 对于高度动态日差异巨大的项性能优化可能不明显、计算成本
